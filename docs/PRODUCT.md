# 筑基插件产品说明

## 一句话

**筑基插件**让 Codex 接入筑基模型、开启插件、使用 AI 生图、拥有长期记忆，并通过飞书在移动端使用。

## 一级入口

| 入口 | 用户看到的名字 | 用户会怎么理解 |
|---|---|---|
| `zhuji-setup` | 接入筑基 | 让 Codex 用上筑基模型，并自动配好推荐参数 |
| `zhuji-codex-plugins` | 开启 Codex 插件 | 让 Codex App 的插件功能可用，同时不改坏登录态 |
| `zhuji-image` | AI 生图 | 在 Codex 里用筑基 `gpt-image-2` 生成图片 |
| `zhuji-memory` | 长期记忆 | 让 Codex 记住用户、项目、最近上下文和重要决定 |
| `zhuji-feishu` | 移动端配置 | 通过飞书在手机上给本地 Codex 发任务和收回复 |

## 为什么这样分

这 5 个入口按用户任务划分，不按技术模块划分。

- `CC Switch`、`fast mode`、长上下文、自动压缩都属于 **接入筑基**。
- 官方 ChatGPT 登录、插件灰屏、插件功能开关都属于 **开启 Codex 插件**。
- 生图只讲 **Codex 里使用筑基 Provider 的 `gpt-image-2`**，不讲操练场。
- 移动端只讲 **飞书入口**，不默认展开公网远程访问。
- 记忆继承 `codex-x` 的本地文件模型，但用户入口只叫 **长期记忆**。

## 默认提示词

插件详情页只放 3 个默认提示词，避免用户选择困难：

1. 帮我接入筑基
2. 帮我开启 Codex 插件
3. 帮我用 Codex 生图

另外两个入口通过用户自然语言触发：

- “帮我创建长期记忆”
- “帮我配置飞书入口”

## 设计边界

### 默认可以做

- 只读检查 Codex 配置和登录状态。
- 生成 CC Switch / Codex 推荐配置。
- 初始化本地记忆工作区。
- 检查飞书桥接配置。
- 解释筑基生图错误。

### 先问再做

- 改 `~/.codex/auth.json` 或 `~/.codex/config.toml`。
- 发起真实生图请求。
- 启动飞书桥接长驻进程。
- 创建 GitHub 用户、创建远端仓库、推送代码。
- 任何公开发布或离开本机的动作。

## 中文命名原则

- 少用英文缩写，除非用户已经知道，比如 CC Switch。
- 入口名字讲“我要做什么”，不是“技术是什么”。
- 详情里再解释 Provider、fast mode、workspace、bridge。
- 遇到风险动作，用中文说清楚备份和回滚。
